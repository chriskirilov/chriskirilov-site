01Problem statement

OM compresses the job-search loop for power-users — every relevant new posting on the open web, scored against a user's actual experience, in a feed they can act on in under five minutes a day.

Same engine doubles as account-level prospecting infrastructure: once you can rank "is this job a fit for this person," you can rank "is this account a fit for our ICP." The architecture is the same; the entities swap.

Operating cadence. ~5,000 (operator, opportunity) evaluations per daily scoring cycle. 12-hour median latency from a posting going live on its source ATS to appearing in the feed. ~600 LLM calls per day at sub-$1 daily inference cost. 88K active jobs, 184K cumulative evaluations, 800+ source boards across 8 ATS platforms. The architectural decisions in the rest of this document are calibrated against that load profile — and against the load profile two orders of magnitude beyond it (covered in §8).

02High-level architecture

Eight stages, registry-driven scrape into a single SQLite database, with the ingestion path feeding both a graph mapper and a Flask API.

Two design commitments worth defending:

  • Registry-driven, not crawler-driven. company_boards is the canonical source of truth for what gets scraped. Every scrape decision is a row, not a hard-coded list.
  • Map-once, score-forever. The expensive LLM work — mapping a user or a job to a skill graph — happens once at ingest. Scoring is deterministic math at query time. This is the cost story I'll defend in §6.

API contract

Flask, Bearer auth via DB_SYNC_API_KEY. Four endpoints, JSON request/response.

EndpointMethodPurpose
/api/tasks/claude/boardGETRanked feed grouped by funnel stage. Returns {stage: [opportunity]}.
/api/tasks/claude/triageGETDaily AI focus pick. 30-min cache. Returns {focus, rationale, opportunities[]}.
/api/tasks/claude/batchPOSTBatched write ops, ≤50 per request. Body: {ops: [{type, opportunity_id, payload}]}.
/api/add-opportunityPOSTManual or extension-driven ingest. Body: {url, ...metadata}. Triggers try_register_board().

Auth is per-route via @login_required, not central middleware. A new endpoint shipped without the decorator ships unauthenticated. I'd move to a Flask blueprint with auth applied at registration before another engineer touches the surface.

Trade-offs

  • SQLite over Postgres. Gives up concurrent writers, gains operational simplicity (single file, cp backup, no daemon). Defended in §7. Inflection point at 10× users, covered in §8.
  • Deterministic graph over vector embeddings. Gives up semantic generalization (an embedding might match "Python" to "scripting" automatically), gains explainability and zero query-time LLM cost. The 186-node graph is hand-curated — feature for explainability, bug at 10K node scale.
  • Map-once over score-on-the-fly. The whole §6 argument. Gives up flexibility (changing the graph requires re-mapping all jobs), gains 250×–1,000× cost reduction.

04Ingestion layer

Eight ATS platforms wired (four populated), plus non-ATS sources, normalized to a uniform job dict.

Sources, populated today:

  • Greenhouse — 824 boards (~49%)
  • Ashby — 590 boards (~35%)
  • Lever — 188 boards (~11%)
  • SmartRecruiters — 65 boards (~4%)

Wired but unpopulated: Workday, Teamtailor, BambooHR, iCIMS — code paths exist via scraper_module dispatch, registry rows = 0.

Supported but not at scale yet. The ingestion code is production-ready for those four; what's missing is the company-discovery sweep populating the registry.

Non-ATS sources: HN, Jobicy, Gmail (LinkedIn / Distill / Notify alerts), browser extension paste, manual /api/add-opportunity.

Schedule (scripts/run_continuous.py):

  • Email ingestion — every 30 min
  • ATS API polling — 6am + 6pm daily
  • Discovery (HN, Jobicy, raw boards) — 6am daily
  • Company-discovery sweep (find new ATS instances) — Sunday 6am weekly

Throughput. ~300–1,000 jobs/day baseline, peak 1,540 on the heaviest day in the last 14.

Normalization. Each ATS client maps its native payload to a fixed dict: {url, title, company, jd_text, ats_type, ats_job_id, location, apply_url, salary_min/max}. Dispatch happens in scripts/run_ats.py keyed on scraper_module. Greenhouse and Ashby return clean JSON; Workday needs {wd, site} params extracted from the URL; iCIMS branches on portal_type (jibe vs classic). The unified dict is platform-agnostic by the time it hits the upsert.

Dedup. Two layers: (1) INSERT OR IGNORE on url UNIQUE handles same-source repeats; (2) event_key deterministic hash dedupes activity rows. Fuzzy cross-source dedup (same job from Greenhouse + manual paste, different URLs) runs offline — that's a known weakness; backups pre-dedup-backup and pre-event-key-backfill mark the last two manual cleanups.

Error handling. scraper.py does 3 retries with exponential backoff on 5xx/403, per-domain 1.5s throttle, ETag/304 conditional GETs on Greenhouse. 429s aren't in the retry set — that's a real bug I'd own up to. Greenhouse uses ETags; the other three don't, so they re-fetch the world daily.

Models. Sonnet 4.5 for quality-sensitive (resume, user mapping), Haiku 4.5 for bulk (scoring, classification, extraction). Prompt caching is on for both (cache_control: ephemeral) — quoting ~90% input-token discount on cached prefixes from Anthropic's pricing.

Batching. Today: per-job synchronous on ingest. Aspirational: move scoring to the Anthropic Batch API for 50% cost off. Currently nothing is batched.

Batch isn't wired yet. Easiest win on the roadmap — daily LLM spend at current operator load is already under $1, so cost pressure hasn't forced the optimization. The 50% Batch API discount kicks in immediately when load grows.

Cost per unit (derived from prompt sizes; would verify against api_usage ledger):

  • Map 1 user (Sonnet, 7K input + 2.5K output) — $0.03
  • Map 1 job (mostly deterministic, ~20% hits Haiku at 3K+1K) — $0.001
  • Score 1 (user, job) — $0, pure math
  • Tailor 1 resume (Sonnet, cached system prompt, 8K output) — $0.05
  • Per-job evaluation_pipeline scoring (Haiku, prompt-cached) — ~$0.005

06Ranking architecture

The core insight: pairwise (user, job) scoring with an LLM is the obvious approach and the wrong one. It's O(users × jobs) LLM calls, unaffordable at multi-user scale. I decompose the problem into map-once, score-forever.

Step 1 — Seed the graph (one-time, system-wide). seed_knowledge_graph() takes the top 50 JDs, runs Sonnet twice (nodes pass + edges pass), produces the 186 nodes / 337 edges that exist today. Two-level hierarchy (L0 cluster, L1 sub-competency). Edge types: composition (child→parent), dependency (A requires B), transfer (A partially advances B with weight 0.3–0.8). Cost: ~$1–2 total, paid once.

Step 2 — Map the user (once per user, on resume change). map_entity_actor(). Sonnet call: takes the resume + profile + the 186-node taxonomy, returns [{node_id, depth 0–100, confidence 0–1, evidence}]. ~80–150 nodes lit per user. Then propagate_skills() runs pure-math edge traversal to infer adjacent skills — propagation at map time, not score time, caches the inference.

Step 3 — Map the job (once per job, on ingest). map_opportunity(). Hybrid 4-stage: rules + fuzzy match + dictionary + LLM-on-residual. Output: [{node_id, required_depth, is_critical, differentiation_power}]. ~15–30 nodes per job. ~80% deterministic, so most jobs cost <$0.001.

Step 4 — Score (every query, pure math). compute_capability_fit(). For each required node: node_score = min(user_depth / required_depth, 1.0), weighted by (type multiplier × differentiation power × IDF rarity). Domain-critical-non-transferable gets 3× weight, gated by user confidence. Coverage penalty if <50% of critical nodes are met. Final scaled to 1–10. Zero LLM calls.

Cost economics — concrete dollars. Take 1,000 users × 5,000 jobs = 5M scorings.

Brute force

Haiku per pair

5,000,000 LLM calls. The obvious approach. Quality also worse — no shared vocabulary across calls.

~$25K — $50K

Graph approach

Map once, math forever

1,000 user maps + ~1,000 job-LLM-residuals + seed. Same ontology for every user/job — scores comparable.

~$40 — $250

That's a ~250×–1,000× delta. The exact figure I'd verify by querying api_usage — that ledger has real token counts per call. Order of magnitude is what matters: it's the difference between an architecture that can't afford to grow and one whose unit economics hold at platform scale.

The deeper point under probing: brute-force pairwise is also worse quality, because the LLM has no shared vocabulary across calls. The graph is a normalized representation — same ontology for every user, every job — which is what makes the scores comparable.

Today's evaluation_pipeline (the 183K rows) is computed by a separate Haiku-per-job dimensional scorer (score_opportunity_fast()). The graph capability score is a second signal feeding the resume tailoring pipeline. I'm in the middle of wiring graph fit into the primary ranking — both signals exist; the graph is feeding resume first, the primary feed scoring is the legacy Haiku-dimensional path.

07Operations

Storage, indexes, observability, failure modes — what's running, what's instrumented, what catches the breaks.

Storage

SQLite, single file (data/jobs.db). The choice is defensible at the system's current load surface. Multi-process write contention is solved by WAL mode plus the activity table's event_key dedup. 88K jobs and 183K evaluations sits well under SQLite's comfortable analytical ceiling. Backup is cp. Migration is a script. The simplicity is the feature — until the load profile changes (covered in §8).

No vector store, no Redis. The "vector" is user_skill_state rows joined against opportunity_skill_requirements rows — relational and indexed. Caching is in-process: _corpus_rarity_cache for IDF, prompt cache (Anthropic-side) for system prompts.

Hot indexes

  • opportunities(url) UNIQUE — drives upsert dedup, every ingest hits this.
  • opportunities(is_active, created_at DESC) — drives the feed query.
  • evaluation_pipeline(user_id, status, adjusted_score DESC) — drives ranked-feed-by-stage.
  • opportunity_activity(event_key) UNIQUE — drives activity dedup.
  • slug_validation_log(slug, validated_at) — drives the 30-day TTL on negative cache.

Heavy read, light write. Feed queries are read-only joins; ingest is the only writer pre-enrichment. Enrichment is the second writer (async, idempotent). At 10× users, the queue worker pool becomes the dominant writer and contention forces the Postgres migration.

Observability

The strongest piece is the api_usage table — every LLM call logged with model, function type, input/output/cache tokens, computed USD cost, batch flag, success flag. get_api_usage_summary() gives 30d/7d/daily breakdowns by function and model. I can answer "how much did the system spend on resume generation last week" with a SQL query.

Beyond that: module-level Python logging across ~15 files. log_activity() persists user-visible events with event_key dedup. pipeline_health() computes a 0–100 user-pipeline health score (applied/interviewing/offer counts, conversion rates, stale apps >7 days).

/health returns hardcoded {status: ok, version} — not a real health check. No central log aggregation, no metrics dashboards, no anomaly detection, no cost circuit breakers. Spend is tracked, not enforced. All on the list.

08What I'd build differently, and where it breaks

The gap between the current operator-scale system and a team-supported production deployment, plus the architectural ceilings that surface as load grows.

What I'd build differently with a real engineering team

This is the section to lean into honesty on, because the senior leader will ask.

  • Tests. 11 test files, none cover scoring, knowledge graph, resume generation, or dedup logic. The graph scorer is the highest-stakes code in the system and has no test coverage. Property-based tests for compute_capability_fit and golden-output tests for the user/job mappers come first.
  • CI. No GitHub Actions, no lint gate, no test gate on PR. Tests run by hand.
  • Schema migrations. All CREATE TABLE IF NOT EXISTS plus ad-hoc backfill scripts. The *.pre-dedup-backup files in data/ are evidence of manual migration moments. Alembic before another engineer touches the schema.
  • Deterministic-vs-AI boundary. Current rule is "rules first, LLM on residual" but the heuristic isn't formalized. Risk: silent quality drift — an LLM confidently mis-mapping a job to nodes that look right but aren't. Offline eval against a golden set for both mappers.
  • Single-process Python. Scoring path is fine. Ingest fan-out (30 concurrent ATS clients) is a ThreadPoolExecutor in one process. Production puts this behind a queue (SQS/Celery) so a crash in one ATS client doesn't take the run down.

Where it breaks at scale

The architecture has two visible walls — both load-driven, both quantifiable, neither yet binding.

First wall — concurrent writes to SQLite. The current single-writer model holds while ingest plus async enrichment is the only contention surface. Once a queue worker pool fans out the ingest path under sustained load, contention exceeds what WAL handles gracefully. Migration is straightforward — Postgres with the same schema, the application layer is already idempotent.

Second wall — LLM-prompt size for user mapping when the graph crosses ~500 nodes. Sonnet's context window is generous, but cost-per-map scales linearly with node count and the prompt cache hit-rate degrades. Mitigation is hierarchical mapping (L0 cluster first, then L1 within matched clusters) to keep the per-call prompt under a token budget. Resume generation crosses a similar wall — at high-volume, batch-processing through the Anthropic Batch API for 50% off, plus prompt-prefix caching across the user's resume context (~$0.05 → ~$0.025 per resume).

Long-tail: slug validation cache. 31K rows today, 95.9% not_found. Linear growth is the negative-cache pattern — at order-of-magnitude growth, the table becomes mostly noise. Bloom filter for the negative path bounds the cache size while preserving the fast-rejection behavior.

Both primary walls are visible from where the system sits today; neither is the gating concern. Observability and a job queue come first — the architectural changes follow load.

Closing argument

Map-once, score-forever.

The architecture exists to convert an O(users × jobs) LLM problem into an O(users + jobs) LLM problem plus pure math. Everything else is plumbing.
